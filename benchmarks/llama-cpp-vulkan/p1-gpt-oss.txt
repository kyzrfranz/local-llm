## Zero‑Downtime Roll‑out Blueprint
**Goal** – Rename a protobuf field (`oldName`) to `newName` while keeping the three services (TS‑React, Go‑REST, Python‑Worker) live and fully functional at every step.

**Assumptions**

| Item | Detail |
|------|--------|
| **Proto compiler** | `protoc` with plugins: `protoc-gen-go`, `protoc-gen-ts` (via `ts-proto` or `protobufjs`), `protoc-gen-python`. |
| **CI/CD** | Each repo has a pipeline that runs unit → integration → contract → smoke tests before a rolling‑deployment (K8s, ECS, or similar). |
| **Versioning** | Semantic versioning for the protobuf package (`vX.Y.Z`). |
| **Feature‑flags** | All services support a simple “useNewField” flag (env var or config). |
| **Observability** | Request/response tracing (OpenTelemetry) and metric `proto_rename_success_total`. |
| **Backward‑compatible protobuf rule** – Adding a new field is safe; renaming is *not*. The rename will be performed as **add‑new‑field + deprecate‑old‑field** → **drop‑old‑field** after all clients have migrated. |

| Step | Owner | Action |

## 2. Proto Change – “Add‑New‑Field + Deprecate”

| Step | Owner | Touch‑point | Action | Verification |
|------|-------|-------------|--------|---------------|
| 2.1 | Proto Maintainer (Go repo) | `shared.proto` | Add the new field with the **same tag number** and mark the old field as `deprecated = true;`. **Do NOT** change the tag number. Example: <br>```proto<br>message MyMessage {<br>  // Old field (will stay for compatibility)<br>  string oldName = 1 [deprecated = true];<br>  // New name, same tag – clients will see the same binary layout<br>  string newName = 1;<br>}``` | Run `protoc --dry-run` and confirm generated files contain **both** `oldName` (deprecated) **and** `newName`. |
| 2.2 | CI | Regenerate code for all three languages (see §3). | Commit the generated files to each repo in a **separate PR** (`proto‑vX+1.0.0`). | CI runs `go vet`, `tsc --noEmit`, `pytest -q` – all must pass. |
| 2.3 | All | Tag the repo with `proto-vX+1.0.0`. | Ensure the tag is **immutable** and published to the artifact registry (e.g., npm, Go module proxy, PyPI). | Pull the new tag in a fresh checkout, run `protoc --version` → confirm generated code matches the schema. |

> **Why this works** – Because the tag number does not change, the wire format is identical; the rename is *only* a source‑level convenience. Existing binaries continue to decode correctly.

## 3. Service‑by‑Service Migration – Order of Dependency

**Dependency graph**

```
React (client) ──► Go REST (API) ──► Python Worker (async consumer)
```

Thus the rollout must flow **upstream → downstream** (client first, then API, then worker) **or** **downstream → upstream** depending on which side writes the field.
In our case **React sends the field**, **Go stores / forwards it**, **Python reads it**.
The safest zero‑downtime path is:

1️⃣ **Go REST** – become *field‑agnostic* (accept both names).
2️⃣ **Python Worker** – read whichever field is present.
3️⃣ **React Frontend** – switch to sending the new name only.
