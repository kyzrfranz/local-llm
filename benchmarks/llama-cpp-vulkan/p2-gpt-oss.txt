# Execution Blueprint
**Goal** – Move a 4 000‑line monolithic Java authentication module into a **Kotlin `auth‑core` library** plus a **thin Java adapter** while keeping **exactly the same runtime behaviour** and **no changes to any of the 30 + existing call‑sites**.

The blueprint is written as a **step‑by‑step, verifiable plan** that can be copied into a JIRA epic or a Confluence checklist.
Every phase has:

* **What to do** (concrete actions)
* **How to verify** (tests, static checks, metrics)
* **Key risks** (what can break)
* **Gate** – “don’t proceed until the verification passes”.

## 2️⃣ HIGH‑LEVEL DECOMPOSITION

| Layer | Responsibility | Current Location (Java) | Target (Kotlin) | Adapter Role |
|-------|----------------|--------------------------|------------------|--------------|
| **Domain / Value Objects** | Immutable data carriers (e.g., `UserId`, `Token`, `AuthResult`). | Mixed throughout the file. | Pure Kotlin `data class`es. | Expose Java constructors / static factories that delegate to Kotlin. |
| **Utilities / Helpers** | String sanitisation, date handling, crypto wrappers. | Static methods. | Kotlin `object`s or top‑level functions. | Java static wrappers that call the Kotlin functions. |
| **Service Interfaces** | `Authenticator`, `TokenProvider`, `CredentialValidator`. | Java interfaces. | Kotlin `interface`s (same JVM signatures). | Extend / implement the Kotlin interface in Java, delegating calls. |
| **Implementations** | `DefaultAuthenticator`, `JwtTokenProvider`, `LdapCredentialValidator`. | Concrete classes. | Kotlin `class`es (internal visibility where possible). | Java adapters that expose the original class names, delegate to Kotlin implementations. |
| **Exception hierarchy** | `AuthException`, `InvalidCredentialsException`. | Java checked/unchecked. | Kotlin `class`es (must stay unchecked to keep binary compatibility). | Java subclasses that simply inherit the Kotlin ones (or re‑export). |
| **Configuration** | `AuthConfig` (properties, builder). | Java POJO + Builder. | Kotlin `data class` + companion builder. | Java builder that forwards to Kotlin constructor. |

**Why this order?**
- **Value objects** have no side‑effects → safest to move first.
- **Utilities** are pure functions → easy to test in isolation.
- **Interfaces** are contracts only → moving them does not affect runtime.
- **Implementations** hold the business logic – moved last after the contract is stable.

## 3️⃣ SAFE EXTRACTION ORDER

The extraction proceeds **incrementally**. After each extraction you **run the full test suite** and **verify the public contract**. If any step fails, you revert, fix, and repeat.

| Step | What to Extract | How to Extract | Verification | Risks |
|------|----------------|----------------|--------------|-------|
| **3.1** | **Package skeleton** – create `auth-core` Gradle module (or Maven sub‑module). Add `kotlin("jvm")` plugin, set `java.sourceCompatibility = 1.8` (or current). | `git mv src/main/java/com/.../Auth.java auth-core/src/main/kotlin/com/.../Auth.kt` (empty stub). | `./gradlew :auth-core:compileKotlin` succeeds. | Build break (classpath, plugin mis‑config). |
| **3.2** | **Value objects** – move each POJO to a Kotlin `data class`. Keep the original Java class as a thin wrapper that **extends** the Kotlin class (or contains a delegate). | Example: `UserId.java` → `UserId.kt` (`@JvmInline value class UserId(val id: String)`). Keep `UserId.java` with `public class UserId extends UserIdKt {}`. | *Compile‑time*: `javac` can still compile the old Java wrapper. <br>*Runtime*: instantiate via Java, `instanceof` works. <br>*Test*: all unit tests that construct these objects still pass. | Nullability mismatches → NPEs. <br>Binary compatibility of inline/value classes (must stay `public`). |
| **3.3** | **Utility functions** – move each static method to a Kotlin top‑level function or `object`. Keep a Java class with static forwarding methods. | `StringUtils.trim(String s)` → `fun String.trimmed(): String = this.trim()`; `StringUtils.java` → `public static String trim(String s) { return StringUtilsKt.trim(s); }`. | *Static analysis*: Detekt passes. <br>*JUnit*: Run all tests that hit the utility. | Kotlin `@JvmStatic` forgetting → No method generated → NoClassDefFoundError. |
| **3.4** | **Exception hierarchy** – rewrite as Kotlin `class`s, keep `java` stubs extending them. | `class AuthException(message: String, cause: Throwable? = null) : RuntimeException(message, cause)`. `AuthException.java` → `public class AuthException extends AuthExceptionKt {}`. | *Throwable check*: `catch (AuthException e)` still works. <br>*Tests*: any test expecting specific exception class passes. | Checked‑exception semantics change (Kotlin always unchecked). Must keep original `extends Exception` if it was checked. |
| **3.5** | **Configuration POJOs** – move to Kotlin data class with `@JvmOverloads` constructor. Add a Java builder that delegates. | `data class AuthConfig(val timeoutMs: Long = 3000, val issuer: String)`; `AuthConfigBuilder.java` → calls `new AuthConfig(timeoutMs, issuer)`. | *JSON/YAML deserialization*: Verify existing deserialisers (Jackson) still produce the same object (add `@JsonCreator`). <br>*Integration tests*: config loading passes. | Jackson Kotlin module missing → fields not bound. |
| **3.6** | **Service Interfaces** – copy to Kotlin unchanged. Add `@JvmDefault` where default methods existed. Keep Java interfaces that **extend** the Kotlin ones (no extra methods). | `interface Authenticator { fun authenticate(req: AuthRequest): AuthResult }` <br>`Authenticator.java` → `public interface Authenticator extends AuthenticatorKt {}` | *Compile*: Java code that implements the interface still compiles. <br>*Tests*: Mocking frameworks (Mockito) can still proxy the interface. | Default methods: need `kotlinOptions { jvmTarget = "1.8" }` + `-Xjvm-default=all`. |
| **3.7** | **Implementation Classes** – move **one class at a time**. For each: <br>1. Create Kotlin version (same package, same class name). <br>2. Keep a Java *adapter* class with the *original* name that **delegates** to the Kotlin class via composition (or inheritance). | `class DefaultAuthenticator : Authenticator { … }` <br>`DefaultAuthenticatorAdapter.java` → `public class DefaultAuthenticator implements Authenticator { private final DefaultAuthenticator delegate = new DefaultAuthenticator(); public AuthResult authenticate(AuthRequest r) { return delegate.authenticate(r); } }` | *All unit tests* for that class run against the Java adapter. <br>*Integration tests* that call the original name (`new DefaultAuthenticator()`) still hit the delegate. | State leakage: static fields in original class must be migrated to Kotlin `companion object` with `@JvmStatic`. <br>Reflection: code that does `Class.forName("com.foo.DefaultAuthenticator")` still works (adapter present). |
| **3.8** | **Dependency Injection bindings** – update the DI container (Spring, Guice, Dagger) to bind the **Java adapter** type (unchanged) to the **Kotlin implementation** (via `@Inject` on delegate). | In Spring XML/Java config replace bean class with `DefaultAuthenticatorAdapter`. In Kotlin implementation add `@Component` (if using Spring). | *Spring context startup* test. <br>*Smoke test*: request an auth token through the whole stack. | Bean lifecycle differences (Kotlin `init` blocks vs Java constructors). |
| **3.9** | **Remove old Java bodies** – once a class is fully covered by Kotlin + adapter, delete the original Java body (keep only adapter). | `git rm src/main/java/com/.../DefaultAuthenticator.java` (after verifying adapter works). | *Full test suite* passes **without** any of the old bodies. | Accidentally delete a class still used elsewhere (search for direct imports). |
| **3.10** | **Finalize the `auth-core` artifact** – publish version `1.0.0‑kotlin` to internal Maven repo. Mark it **binary compatible** (use `java -jar` `japicmp` to compare with previous JAR). | Run `./gradlew :auth-core:publish` and `japicmp` diff. | *Compatibility report* shows **no** public API changes (except added `*Adapter` classes). | If a downstream module uses reflection to access private members, compatibility breaks. |

> **Gate after each Step** – run the *Full Test Suite* (unit + integration + contract + performance). If any test fails, **stop** and fix before proceeding.

## 5️⃣ RISK MATRIX & MITIGATION

| # | Risk | Where it appears | Impact | Likelihood | Mitigation |
|---|------|-------------------|--------|------------|------------|
| R1 | **Nullability mismatch** – Kotlin treats parameters as non‑null by default, Java callers may pass `null`. | Value objects, utility functions. | NPE at runtime. | **High** | • Annotate Kotlin parameters with `@Nullable` where needed.<br>• Add explicit `requireNotNull` checks and unit tests with null inputs.<br>• Run `nullaway` on Java code after migration. |
| R2 | **Binary compatibility break** – downstream modules use reflection (`Class.forName`, `Method.invoke`). | Any moved class. | `NoSuchMethodError`/`ClassNotFoundException`. | **Medium** | • Keep the original **Java adapter** class name & package unchanged.<br>• Run reflection‑based integration tests (e.g., Spring’s `ClassPathScanningCandidateComponentProvider`). |
| R3 | **Kotlin default methods** – Java code expects default implementations that Kotlin does not emit without `-Xjvm-default`. | Service interfaces with default methods. | AbstractMethodError. | **Low** | • Enable `kotlinOptions { freeCompilerArgs += "-Xjvm-default=all" }`.<br>• Add explicit `@JvmDefault` on each default method. |
| R4 | **Serialization incompatibility** – Jackson/ Gson mapping to old POJOs. | Config objects, DTOs. | Missing fields, `InvalidDefinitionException`. | **Medium** | • Add `@JsonCreator` and `@JsonProperty` on Kotlin constructors.<br>• Keep a Java DTO wrapper if needed (delegates to Kotlin). |
| R5 | **Dependency‑injection mismatch** – Spring proxies Java classes but not Kotlin final classes. | Implementation classes. | Bean creation failure. | **Low** | • Mark Kotlin classes as `open` (or use `kotlin-spring` plugin to make them open automatically). |
| R6 | **Performance regression** – Kotlin generated bytecode may be slower for crypto primitives. | Crypto helpers, token signing. | Higher latency. | **Low** | • Benchmark critical paths (JMH) after each implementation migration.<br>• Keep hot‑path methods in Java if regression observed (allow mixed language). |
| R7 | **Build‑pipeline breakage** – Gradle/Maven not configured for mixed Java/Kotlin. | Project setup. | No CI builds. | **Medium** | • Add separate source sets (`src/main/kotlin` & `src/main/java`).<br>• Verify that `clean build` works locally before pushing. |
| R8 | **Version drift** – The Java adapter and Kotlin core get out‑of‑sync in a multi‑branch environment. | Release management. | Runtime mismatches. | **Low** | • Publish both artifacts together with the same version.<br>• Use a *bom* (Bill‑of‑Materials) to lock versions. |
| R9 | **Testing blind spot** – Existing tests only cover happy path; hidden edge‑cases exposed by Kotlin null‑safety. | All phases. | Undetected bugs. | **Medium** | • Add *property‑based* tests (Kotlin `kotest` `property`) for key functions.<br>• Run mutation testing to surface missing branches. |
| R10 | **Team knowledge gap** – Developers unfamiliar with Kotlin may introduce bugs in later steps. | Ongoing development. | Slower progress, quality drop. | **Medium** | • Provide a short “Kotlin for Java devs” workshop.<br>• Enforce code‑review checklist that includes “Kotlin interop sanity”. |
