### Strict Diagnostic Blueprint: Isolating a Hang in a New Integration Test After Patch Application

This blueprint prioritizes **actionable, low-effort checks first**, escalating to complex diagnostics only when necessary. It assumes a production-grade environment (e.g., Linux/Unix-based systems; Windows equivalents provided where relevant). All steps are designed to be executed sequentially with minimal setup.

---

#### **Step 0: Confirm It's a Hang (Not a Failure)**
**Why?** A "hang" means the process is *stuck indefinitely without exiting or producing output*. A failure would complete with an error/stack trace.
**How to isolate:**
1. **Check process state**:
   - Run `ps aux | grep <test_process_name>` (Linux) or `Get-Process -Name <name>` (Windows).
     - If `STAT` shows `D` (uninterruptible sleep) or `S` (sleeping) with **0% CPU usage** → **likely a hang**.
     - If `R` (running) with **high CPU usage** → likely an infinite loop (still a hang variant).
     - If the process exited (no entry in `ps`/`Get-Process`) → it failed; check logs for errors.
2. **Check output**:
   - Run the test alone (not whole suite) with `> test.log 2>&1`.
     - If the log file is **empty** and the process remains running → **confirmed hang**.
     - If the log has partial output but no completion → **hang**.
     - If the log shows an exception/stack trace → **failure** (not a hang).
3. **Force stack trace** (critical for confirmation):
   - **Java/Kotlin**: `jstack <PID>` → look for "waiting on" threads with no progress.
   - **Python**: `kill -SIGUSR1 <PID>` (requires `faulthandler` enabled) or `gdb -p <PID> -ex "thread apply all bt"`.
   - **Go**: `kill -SIGQUIT <PID>` → check goroutine dumps for blocked channels.
   - **C/C++**: `gdb -p <PID> -ex "thread apply all bt"`.
   - **Result**: If stack traces show **no active threads** or **all threads blocked on locks/I/O**, it's a hang.

> ✅ **Decision point**: If this step confirms a hang, proceed to Step 1. If it’s a failure, stop—this blueprint doesn’t apply.

#### **Critical Actions for Each Path**
- **Deadlock (Path C → G → K)**:
  - Search the patch for `synchronized`, `mutex.lock()`, `ReentrantLock`, or similar.
  - Check if locks are acquired in inconsistent orders (e.g., Thread A locks X then Y, Thread B locks Y then X).
  - **Quick test**: Comment out all synchronization in the new code. If the test passes → deadlock confirmed.

- **Blocking I/O (Path D → H → L)**:
  - Look for `http.get()`, `socket.accept()`, `file.read()`, or DB queries in the patch.
  - **Quick test**: Add explicit timeouts (e.g., `requests.get(url, timeout=5)` in Python; `setSoTimeout(5000)` in Java). If it times out → I/O issue.

- **Infinite Loop (Path E → I → M)**:
  - Search for `while(true)`, `for(;;)`, or recursive calls without base case in the new test/production code.
  - **Quick test**: Add `print("loop iteration")` inside loops. If output stops at a specific point → loop issue.

- **External Dependency (Path F → J → N)**:
  - If the test calls an API, DB, or file system:
    - Manually verify the resource exists (e.g., `psql -c "SELECT 1"` for DB).
    - Replace the dependency with a stub that returns immediately (e.g., mock HTTP server returning 200).
  - **Quick test**: Run the test against a local dev instance of the dependency. If it passes → issue is external.

- **Teardown Issues (Path Q → R)**:
  - Check `@After`, `tearDown()`, or cleanup code in the test.
  - **Quick test**: Comment out the entire teardown section. If the test runs → teardown is hanging.

#### **Why This Order?**
1. **Stack traces are definitive**: They directly show where threads are stuck (e.g., locks vs. I/O), making this the fastest way to rule in/out hypotheses.
2. **Low-effort checks first**: Adding timeouts or mock replacements takes <5 minutes and often resolves the issue.
3. **Patch focus**: The new test is the only change—90% of hangs stem from code added by the patch. Start there before broad system checks.
4. **No wasted time**: Skipping full debugging tools (e.g., `strace`, `perf`) until evidence demands it.

> 💡 **Final Tip**: If all else fails, **revert the patch** and reapply it incrementally. Isolate which specific change triggers the hang using binary search on the diff. This is the ultimate fallback for complex patches.
